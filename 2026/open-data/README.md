## 準備するもの

- ノートPC（Windows / Mac どちらでもOK）
- モダンブラウザ（Chrome 推奨）

### ローカルサーバーで開く（推奨）

HTML を **ファイルとして直接開く**（`file://`）と、ブラウザのセキュリティ制限で地図タイルの読み込みなどが失敗することがあります。**`http://localhost` で開く**と安定します。

| 方法 | 手順 |
| :-- | :-- |
| **VSCode Live Server** | 拡張機能「Live Server」を入れ、`index.html` を右クリック → **Open with Live Server** |
| **Python** | ターミナルで `2026/open-data` に移動し、`python3 -m http.server 8080` → ブラウザで `http://127.0.0.1:8080/` を開く |
| **Node.js** | `npx --yes serve . -p 8080`（同様に `http://127.0.0.1:8080/`） |

授業では Live Server を使う前提で進めるとスムーズです。

## この授業のゴール

**「身の回りの課題」→「オープンデータで解決できそう」→「Web アプリで解決する」** という流れを体験する。

プログラミングの技術は「課題を解くための道具」として、必要になったタイミングで登場します。

前半は身近な困りごとから問いを立て、後半は [Code for FUKUI](https://github.com/code4fukui) の公開データを使って実装します。最後は公開 API も組み合わせて、アプリをさらに広げます。

## 作るもの：福井恐竜博物館 おでかけプランナー

### なぜ作るのか（課題）

> 「GW に恐竜博物館に行きたいけど、混みそうで迷ってる。いつ行けば空いてる？ 近くに他にどんな観光スポットがある？」

福井県立恐竜博物館は世界三大恐竜博物館のひとつ。でも GW や夏休みは大混雑。  
「混んでいるのはわかっているけど、具体的にいつが空いているか」は見えにくい。

### オープンデータで解決できそう

[Code for FUKUI](https://github.com/code4fukui) が 2 つのオープンデータを公開している。

| データ | 内容 | 形式 |
| :-- | :-- | :-- |
| [dinosaur-opendata](https://github.com/code4fukui/dinosaur-opendata) | 恐竜博物館の 60 日先までの予約人数・金額（毎日更新） | CSV |
| [fukui-spot](https://github.com/code4fukui/fukui-spot) | 福井県の観光スポット一覧（名前・ジャンル・位置情報） | CSV / JSON |

### 完成アプリのイメージ

- 予約人数を棒グラフで表示 →「GW のどの日が混んでいるか」が一目でわかる
- 「空いている日」を色で強調 →「いつ行くか」の判断材料になる
- 観光スポット一覧をジャンルで絞り込み →「行く前後に寄れる場所」がわかる

## Day 1（約3時間・計185分）― 問いを形にする日

スライドでは **パート 1〜5**（開発の準備 → 問い → Chart.js → fetch）までを Day 1 とする。

### パート 1：VSCode を使ってみよう ― 開発の準備をする（15分）

[Visual Studio Code](https://code.visualstudio.com/) に慣れ、今日の実習の土台を作る。

- 拡張機能（例：[Live Server](https://marketplace.visualstudio.com/items?itemName=ritwickdey.LiveServer)、Prettier、日本語パック）の導入
- **[`template/`](./template/) をコピーしたフォルダ**でハンズオンを始める（詳細は `template/README.md`）
- `index.html` / `styles.css` / `app.js` を開き、保存と Live Server でのブラウザ確認の流れ
- 表示がおかしいときはエディタとブラウザの開発者ツール（Console）を見る

### パート 2：Web の三本柱 ― HTML・CSS・JavaScript（15分）

Web ページは 3 つの技術で成り立っている。

| 技術 | 一言で言うと | 担当する仕事 |
| :-- | :-- | :-- |
| **HTML** | 骨格 | 「何が」あるかを定義する（見出し・段落・ボタン・表など） |
| **CSS** | 見た目 | 「どう見せるか」を決める（色・大きさ・レイアウト・アニメーションなど） |
| **JavaScript** | 動き | 「どう動くか」を書く（ボタンを押したら動く、データを取ってくる、計算するなど） |

- 今日作るアプリのファイル構成（`index.html` / `styles.css` / `app.js`）を確認する
- ブラウザの開発者ツール（F12）で HTML 構造・`<canvas>`・Console を眺める
- 「今日書くコードのどこが HTML でどこが JS か」を意識しながら進める

### パート 3：問いを立てる ― 身近な問題から考える（45分）

まずは「どんなデータがあるか」ではなく、**自分の身の回りで困っていること**から始める。

- 「混み具合がわからない」「行くタイミングを決めにくい」「比較しづらい」などの困りごとを各自で出す
- その困りごとが「データがあれば判断しやすくなるか」を考える
- 完成デモを見て、「問いをアプリで解く」とはどういうことかをつかむ

全員の問いをそのまま実装するのは難しいので、[Code for FUKUI](https://github.com/code4fukui) の公開データを使って、**近い形で試せる問い**に絞る。

- [dinosaur-opendata](https://github.com/code4fukui/dinosaur-opendata) と [fukui-spot](https://github.com/code4fukui/fukui-spot) を見て、「どんな判断ができそうか」を考える
- 「いつ行けば空いているか」「行く前後にどこへ寄れるか」という問いに落とし込む
- 「このデータがあるからアプリが作れる」という見通しを持つ

### パート 4：Chart.js でグラフ ― データを見える化する（55分）

「数字の羅列では判断できない」から、グラフが必要になる。ここで Chart.js が登場。

- HTML ファイルの基本構成（`<canvas>` タグ・`<script>` で Chart.js をインターネットから読み込む）
- ハードコードしたデータで棒グラフを表示してみる
- `data` / `labels` / `datasets` の構造を理解する
- 色・タイトル・ツールチップのカスタマイズ
- **ハンズオン：** サンプルデータで「予約人数の棒グラフ」を表示する

### パート 5：fetch と async/await ― データを自動で取得する（55分）

> 「毎回 CSV を手で貼り付けるのは大変。自動で最新データを取ってきたい」

ここで `fetch` と `async/await` が登場。

- `fetch` の仕組み（URL を渡すとデータが返ってくる）
- `async` / `await` とは何か（「データが来るまで待つ」）
- GitHub Raw の URL から CSV を取得する
- CSV を JavaScript の配列に変換する（`split('\n')` / `split(',')` / `map`）
- **ハンズオン：** 恐竜博物館の最新 CSV を fetch で取得して棒グラフに表示する

## Day 2（約3時間・計175分）― 機能を広げる日

スライドでは **パート 6〜9** を Day 2 とする。Day 1 で予約データのグラフが最新化できた前提で、スポット一覧・色分け・公開 API・改善に進む。

### パート 6：観光スポット一覧 ― DOM 操作と filter（45分）

> 「博物館の混雑はわかった。せっかく行くなら近くのスポットも知りたい」

fukui-spot データを使い、2 つ目のデータを追加する。

- 観光スポットの CSV を読み込む（教材では Papa Parse を利用）
- `createElement` / `innerHTML` でスポットカードを画面に並べる
- `<select>` のドロップダウンでジャンル（自然・食・歴史など）を絞り込む
- `filter` で条件に合うスポットだけ表示する
- **ハンズオン：** ジャンルを選ぶとスポット一覧が絞り込まれる機能を作る

### パート 7：色分けと TOP 3 ― 直感的な UI を作る（30分）

> 「グラフだけでなく、ひと目で『この日が空いている』とわかるようにしたい」

- 予約人数のしきい値（例：500人未満）で「空いている／混んでいる」を判定する
- Chart.js の `backgroundColor` を条件分岐で色分けする（混雑日は赤・空き日は緑）
- 「おすすめ日 TOP 3」をテキストで表示するロジックを書く
- **ハンズオン：** 混雑度に応じてグラフのバーの色が変わるようにする

### パート 8：公開 API でおすすめ日を作る ― CSV のつぎに JSON を足す（40分）

オープンデータ（CSV）は `fetch` で取り込めた。次に「天気」のように **CSV にない情報** を足したいとき、**公開 API**（URL にアクセスすると JSON が返る）へ同じ `fetch` でつなげる、という流れです。

- [Open-Meteo](https://open-meteo.com/) などの公開 API を `fetch` で取得し、JSON データを読む
- 「晴れそうな日 × 空いている日」のように、複数データを組み合わせる発想を体験する
- 予約人数と天気をもとに「おすすめ日スコア」を考える

### パート 9：改善タイム・発表 ― グループ内で共有（60分）

- 各自で改善テーマを 1 つ決めて手を入れる
- 例：Leaflet で地図表示、`localStorage` でお気に入り保存、しきい値や表示文言の改善
- **グループ内で発表**：目的・使った技術・どこまで進んだかを短く共有する（ブラウザでデモできればベスト）

## 技術の登場タイミングまとめ

| 技術 | 登場する文脈（なぜ必要になるか） |
| :-- | :-- |
| Chart.js | 「数字の羅列では判断できない」からグラフが必要になる |
| `fetch` / `async/await` | 「毎回手でデータを貼るのは大変」だから自動取得したくなる |
| CSV パース（`split` / `map`） | 「取ってきたデータを使えるデータに変えたい」から |
| DOM 操作（`createElement`） | 「スポットを画面に並べたい」という表示要求から |
| `filter` | 「全部じゃなく条件に合うものだけ見たい」という操作要求から |
| 条件分岐 + 色設定 | 「数値ではなく直感的に判断できるようにしたい」という UX 要求から |
| 公開 API / JSON | 「CSV だけでは足りない情報を足したい」から（同じ `fetch` で JSON を受け取る） |
